# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

KOPASEBA (Koperasi Paru Sejahtera Bahagia) — a full-stack marketing + product catalog site for an Indonesian cooperative owned by PDPI (Perhimpunan Dokter Paru Indonesia). Originally a static site; now a database-backed Next.js app with a public catalog and an admin dashboard for managing all site content. All page content is in Indonesian. Deploys to **Hostinger Node.js Web App Hosting** — see `DEPLOYMENT.md` for the full deploy runbook.

## Commands

```bash
npm run dev        # start dev server (Next.js, Turbopack default)
npm run build      # prisma generate && next build
npm run start      # prisma migrate deploy && next start (production)
npm run lint       # eslint (flat config, eslint-config-next)
npm run db:seed    # tsx prisma/seed.ts — idempotent, safe to re-run
npm run db:studio  # prisma studio — browse/edit the database directly
```

There is no test suite configured in this repo.

**Local database**: MySQL via Docker, not a local MySQL install. Container: `docker run -d --name kopaseba-mysql -e MYSQL_ROOT_PASSWORD=kopaseba_dev -e MYSQL_DATABASE=kopaseba -p 3307:3306 mysql:8.4` (port 3307 host-side, since 3306 is commonly taken locally). `.env`'s `DATABASE_URL` points at it with the `root` user — `migrate dev` needs shadow-database privileges that a scoped app user won't have.

## Stack

- Next.js 16 App Router, React 19, TypeScript (strict), Tailwind CSS 4.
- **Prisma 7** with the newer `prisma-client` generator (not `prisma-client-js`) — this means: a mandatory **driver adapter** (`@prisma/adapter-mariadb` + `mariadb` package for MySQL), client generated to `src/generated/prisma/` (gitignored, regenerated by `prisma generate`), and `prisma.config.ts` (not `schema.prisma`) holding the datasource URL. Import the client via `@/generated/prisma/client` and `@/generated/prisma/enums`, not `@prisma/client`. See `src/lib/prisma.ts` for the singleton pattern.
- **Auth.js v5** (`next-auth@beta`), Credentials provider, JWT session strategy (the Credentials provider does not support database sessions — this is a library constraint, not a choice). See "Auth" below.
- **shadcn/ui** on the **Base UI** primitive flavor (`@base-ui/react`, not Radix) — components use a `render` prop for custom elements (e.g. `<Button render={<Link href="..." />} />`), not `asChild`. `Button`/`SidebarMenuButton` etc. default `nativeButton: true`; pass `nativeButton={false}` when rendering as a non-`<button>` element (e.g. a `Link`) or Base UI logs a console warning.
- Tailwind is configured via CSS (`src/app/globals.css`), not a `tailwind.config.*` file — theme tokens live in `@theme inline` there. shadcn's generated `--primary`/`--secondary`/etc. variables are remapped in the same file to the brand palette (see "Design tokens" below) rather than left at shadcn's defaults.
- Path alias `@/*` → `./src/*` (see `tsconfig.json`).
- Next.js middleware is named `src/proxy.ts` (Next 16 renamed the `middleware` file convention to `proxy`; `middleware.ts` is deprecated). Proxy defaults to the Node.js runtime, which is required here since it calls `auth()` → Prisma, and Prisma's MySQL driver adapter isn't Edge-compatible.

## Architecture

### Route groups

- `src/app/(site)/` — all public marketing/catalog pages (`page.tsx` home, `tentang`, `pengurus`, `unit-usaha`, `program-kerja`, `keanggotaan`, `legalitas`, `kontak`, `katalog`, `katalog/[slug]`), wrapped by `(site)/layout.tsx` which renders `Navbar` + `Footer`. The root `src/app/layout.tsx` only sets up `<html>`/fonts/metadata — it does **not** render Navbar/Footer, so admin routes don't inherit the public chrome.
- `src/app/admin/login/` — standalone login page, no sidebar.
- `src/app/admin/(dashboard)/` — everything behind auth (`katalog`, `pengurus`, `unit-usaha`, `legalitas`, `program-kerja`, `keanggotaan`, `tentang`, `kontak`, `akun`), wrapped by a layout that renders the sidebar shell and re-checks the session server-side (defense in depth on top of the proxy/middleware check).
- `src/proxy.ts` gates `/admin/:path*` (redirects to `/admin/login` if unauthenticated; redirects away from `/admin/login` if already authenticated).

### Data layer

Every content type that used to live in `src/data/content.ts` is now a Prisma model, editable via the admin dashboard. **`content.ts` now exports only `nav`** — the header/footer link structure — deliberately kept hardcoded rather than DB-driven (it's an IA/design decision, and making it admin-editable risked an admin fat-fingering navigation, including locking themselves out of `/admin/login`'s own link path). `Navbar` uses its own trimmed `navPrimary` array (not `content.ts`'s `nav`); `Footer` uses the fuller `nav` from `content.ts` as the sitemap. Keep these two lists intentionally different in scope, not synced.

Key models (`prisma/schema.prisma`):
- `KatalogItem` — public product/service catalog (`kategori`: `PRODUK_MITRA` | `LAYANAN_UNIT_USAHA`), with `harga` + a separate `tampilkanHarga` boolean so price can be stored without being shown publicly.
- `Pengurus` — unifies what used to be four separate arrays (`pimpinanUtama`/`dewanPengawas`/`anggotaPengurus`/`sekretariat`) into one model with a `kategori` enum.
- `Media` — generic image blob storage (`data Bytes @db.LongBlob`) for katalog and pengurus photos, served via `/api/media/[id]/route.ts` with long-lived `Cache-Control`. Images are stored **in MySQL, not on disk** — deliberate, because Hostinger's redeploy behavior for untracked filesystem writes isn't guaranteed, while the database is. Replacing a photo inserts a new `Media` row rather than mutating bytes in place.
- `ListItem` — polymorphic table (`section` enum) backing every plain bullet-list on the site (syarat/hak/kewajiban/prosedur anggota, landasan prinsip, fungsi peran, program kerja jangka pendek/panjang/skema bagi hasil). One generic admin component (`ListEditor`) drives all of these instead of building a bespoke CRUD screen per list.
- `SiteSettings` — a singleton row (`id` fixed at `1`) holding scattered scalar text: kontak info, tentang prose (landasan/asas/tujuan), program-kerja section titles.
- `KeanggotaanFAQ`, `SejarahTimeline`, `LegalitasDoc`, `StrukturTambahan`, `UnitUsahaPillar` — each a straightforward CRUD model for one page section.
- `User` — admin accounts. `sessionVersion` is bumped on password change or "log out everywhere"; the JWT callback re-checks it against the DB on every request, so bumping it invalidates all previously-issued sessions without needing a database session store.

`prisma/seed.ts` is a thin CLI wrapper around `src/lib/seed.ts`'s `runSeed()`, which is also called by `src/app/api/seed/route.ts` (a token-gated HTTP fallback for environments without SSH access — see `DEPLOYMENT.md`). Every seed step is idempotent (guarded by a `count() === 0` check per table, or `upsert` for the admin user), so re-running it never duplicates or clobbers admin-edited data.

### Server Actions

Mutations live in `src/lib/actions/*.ts` (one file per domain: `katalog.ts`, `pengurus.ts`, `legalitas.ts`, `unitUsaha.ts`, `kontak.ts`, `tentang.ts`, `programKerja.ts`, `listItem.ts`, `faq.ts`, `sejarah.ts`, `auth.ts`, `media.ts`), not co-located under `app/admin/**`. Every mutating action starts with `await requireAdmin()` (`src/lib/require-admin.ts`) — this is the actual authorization boundary, since Server Actions are directly POST-able and the proxy/middleware check alone isn't sufficient. Reads are plain `await prisma.x.findMany(...)` calls directly in Server Components, not Server Actions.

Reordering (`urutan` field) is done via up/down swap actions (`reorderXAction(id, ..., "up" | "down")`) that swap `urutan` between adjacent rows in a `$transaction`, rather than drag-and-drop — simpler to get right, no client-side DnD library. New rows get `urutan` computed server-side as `count()` of existing siblings (append to end); don't trust a client-submitted `urutan` on create.

After every mutation, call `revalidatePath()` for both the admin list and the corresponding public route(s) (and `/` when the change could affect the homepage's pimpinan/katalog-teaser/program-kerja sections) — there's no ISR polling here, public pages reflect edits immediately because they're `export const dynamic = "force-dynamic"` Server Components reading Prisma directly.

### Auth

`src/lib/auth.ts` configures Auth.js with a single Credentials provider (bcryptjs password check, brute-force lockout after 5 failed attempts within `User.lockedUntil`), JWT session strategy, and a `jwt` callback that re-fetches `sessionVersion` from the DB on every request to support forced logout. Module augmentation for the JWT type must target `@auth/core/jwt`, not `next-auth/jwt` — the latter is a re-export and TypeScript's `declare module` doesn't resolve through it under this project's `moduleResolution: "bundler"`.

### Admin UI conventions

- shadcn/ui components in `src/components/ui/`, admin-specific composites in `src/components/admin/` (`KatalogForm`, `PengurusForm`, `ListEditor`, `FAQEditor`, `SejarahEditor`, table components, etc.) — keep this split; don't add admin-only components to `src/components/ui/`.
- Forms use `useActionState` bound to a Server Action (`action.bind(null, id)` for edit-mode), plain `<Field>`/`<FieldGroup>`/`<FieldLabel>`/`<FieldError>` layout (shadcn's Field primitives), not React Hook Form — the forms here are simple enough that server-side zod validation + `useActionState`'s returned error is sufficient.
- Delete confirmations use `AlertDialog`; add/edit-in-place dialogs (used by `ListEditor`, `FAQEditor`, `SejarahEditor`) use `Dialog` with local `useState` + `useTransition` calling the action directly (not a form action) since they need to close the dialog only on success.
- Zod schemas live in `src/lib/validation/*.ts`, one per domain, mirroring the actions split.

## Design tokens

`src/app/globals.css`: `paper`/`paper-dark` (background), `ink` (text), `navy`/`navy-dark`/`navy-light` (primary), `sky`/`sky-light` (accent), `gold`/`gold-dark`/`gold-light` (accent/CTA). Fonts: `font-serif` = Fraunces (headings), `font-sans` = Inter (body) — both loaded via `<link>` tags in `src/app/layout.tsx` head, **not** `next/font` (the `@next/next/no-page-custom-font` lint rule is deliberately disabled for this in `eslint.config.mjs`). If you ever re-run `npx shadcn add`/`init`, check that it hasn't reintroduced a `next/font` Geist import in `layout.tsx` or reset `--font-sans` in `globals.css` — it has done this before.

shadcn's semantic tokens (`--primary`, `--secondary`, `--muted`, `--sidebar`, etc.) are mapped onto the brand palette in the same `:root`/`.dark` blocks in `globals.css`, not left at shadcn's default zinc/oklch values — the admin sidebar is intentionally navy with gold active-state highlighting to match the public site rather than looking like a generic shadcn dashboard.

Layout convention across public pages: `max-w-6xl mx-auto px-6` containers, alternating full-bleed `bg-navy`/`bg-sky-light`/plain sections, rounded-2xl/3xl cards with `border border-ink/10`. Shared components (`PageHeader`, `CTASection`, `IconBadge`, `Timeline`, `DocumentCard`, `MapEmbed`, `KatalogImage`, `PersonPhoto`) live in `src/components/` — reuse these rather than hand-rolling equivalent markup on a new page. `PersonPhoto` and `KatalogImage` both accept a nullable media src and render a placeholder when absent (no photo uploaded yet) — follow this pattern for any new image-backed content type instead of assuming a photo always exists.

Every inner public page ends with a `<CTASection>` linking to 1-2 related pages — this is the site's primary cross-navigation mechanism now that the header (`Navbar`) only shows 5 top-level items (Beranda/Katalog/Unit Usaha/Tentang/Kontak) plus the "Gabung Jadi Anggota" CTA. Struktur Organisasi, Program Kerja, Keanggotaan, and Legalitas are intentionally not in the header — they're reachable via `CTASection` links and the Footer's full sitemap instead. Keep this in mind before adding a new public page: decide its `CTASection` links rather than assuming it needs a header entry.

`design-reference/` is not part of the built app — it holds source material used when writing/updating site content and design (mockup HTML, the sosialisasi PPTX, the official SK PDF, the AD-ART bylaws doc). Check these when content needs verifying against the primary source rather than guessing.
